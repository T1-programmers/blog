### 관계형 데이터 모델링
## 전체 흐름

#### 문제를 현실로부터 뜯어내서 고도의 추상화 과정을 거쳐 데이터베이서로 옮겨 담는 작업

→개념적 데이터 모델링  : 내가 하고자 하는 이 일에 어떤 개념적 데이터가 있고, 어떻게 연결되어 있는가

→ 논리적 데이터 모델링 : 패러다임에 맞는 표로써 생각했던 개념을 표로 전환하는 작업

→물리적 데이터 모델링 : 어떤 데이터베이스 서비스를 사용하고, 최적화된 코드를 작성해서 데이터를 보관

## 개념적 데이터 모델링

관계적 데이터 모델링 중에 제일 중요한 부분

개념적 데이터 모델링이 잘 되어 있다면,

논리적, 물리적 데이터 모델링을 수월하게 진행할수 있다.

**ERD (Entity relationship diagram)**

정보를 발견하고, 연관된 정보들끼리 그룹화하고,

그룹화 한 정보들을 표현할수 있는 개념

## 관계형 데이터 베이스 다운 개념의 구조

설명이 가능하고, 표현이 가능하다면 문제가 없다.

그러나 RDB는 내포관계를 허용하지 않는다.(틀린것은 아니지만 , 한계가 있다)

거대 단일 테이블로 표현을 하면, 중복이 발생한다.

표를 분리해서 관리하게 되면, 주제에 따라서 데이터를 효율적으로 관리 할 수 있다. Join 이라는 명령어 알고리즘으로 필요한 데이터를 추출함에 있어서 데이터 자원을 아낄 수 있다.

## ERD의 구성 요소

Entity 는 구체적인 정보들의 관계를 담은 추상적인 최소 단위의 파일만 담을수 있는 폴더같은 개념이다.

이후 논리적 데이터 모델링때 오른쪽 개념으로 변경된다.

Entity = Table

Attribute = Column
Relation  = PK, FK

## Entity 의 정의

각 Entity를 이루고 있는 속성을 정의하고,

identifier(식별자)를 선정해야 한다. (Primary key, Unique key)

(그 대상만 고유하게 가지고 있는 값을 가져야 한다)

식별자가 가지고 있는 값을 통해 다른 속성과 연결된다.

## Cardinality / Optionality

Cardinality

특정 Entity의  한 속성의 다른 Entity 속성과의 1:1 , 1:N , N:M의 관계성 (한쪽의 데이터가 다른 속성의 몇개와 연결되는가)

Optionality

특정 Entity의 한 속성이 다른 Entity 속성과 매칭 여부{Mandatory/Optional}, (반드시 매칭되는 데이터가 있어야 하는가)

독자적으로 존재할수 있는 Entity는 부모 Entity

필수적으로 매칭되어야 존재 가능한 Entity는 자식 Entity

## 논리적 데이터 모델링

관계형 데이터 모델링에 패러다임에 맞게 데이터 형식을 정리하는 단계

Mapping Rule :ERD로 정리한 개념을 논리적 데이터 모델링에 맞게 프로그램에 재정리 하는 단계

Entity → Table

Attribute → Column
Relation →PK, FK

1:1 , 1: N 관계에서의
PK, FK는 부모 테이블이 자식 테이블에게 준다.

N:M 의 관계에서는 서로 PK를 넘겨주기 어렵기 때문에

Mapping Table(연결 테이블)을 만든다.
